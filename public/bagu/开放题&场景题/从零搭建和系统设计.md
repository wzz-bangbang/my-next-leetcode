##### 如果要做一个拖拽式仪表盘Dashboard用户能自己拖拽、组合图表组件展示不同数据源的信息，你会怎么来设计这个系统

我会把这个系统拆成四层：**布局层、组件层、数据层、权限与发布层**，保证既能拖拽搭建，又能稳定扩展。
 1) 布局层（Layout）
- Dashboard 的核心产物是一个 **Layout Schema**：  
    记录每个卡片（Widget）的 `id、位置(x,y)、尺寸(w,h)、层级、响应式断点`。
- 前端用成熟的栅格拖拽方案（类似 react-grid-layout 这种思想）：拖拽只改 layout，不直接影响数据逻辑。
- 支持：对齐/吸附、最小尺寸、断点布局（desktop/tablet/mobile）。
 1) 组件层（Widget/图表物料）
- 每个图表是一个可注册的 Widget：  
    `meta(配置面板schema) + render(运行时组件) + dataAdapter(数据适配器)`。
- 配置面板是 schema-driven：用户选指标、维度、过滤器、样式；平台把它存成 `widgetConfig`。
1) 数据层（Data Gateway + Query Model）
- 性能策略：
- 请求合并：同一数据源相同 query 做 dedupe
- 缓存：按 query key 缓存 + TTL
- 并发控制与取消：切过滤器时 cancel 旧请求
- 大表/高频：优先用聚合接口，必要时异步查询（返回 jobId 轮询）
 1) 权限与发布
- 权限到 dashboard / widget / 数据源 / 字段（必要时行级）。
- 支持“私有/团队共享/公开”，公开也走 token + 限流 + 脱敏策略。
- 可观测：每个 widget 的 query 耗时、失败率、命中缓存率，方便定位哪个卡片拖慢整页。

这个系统的关键是把复杂度隔离：**布局只是布局、组件只是展示、数据统一走网关、权限和审计在服务端兜底**，这样才能扩展物料和数据源且不失控。
