####  CSR-SSR-SSG-ISR
##### 1️⃣ CSR（Client-Side Rendering，客户端渲染）
- HTML 基本是空壳
- JS 下载完成后在浏览器渲染页面

**特点**
- 首屏慢（白屏时间长）
- SEO 不友好（需额外方案）
- 交互流畅（路由切换快）

**适用场景**
- 后台系统
- 强交互应用（B 端、管理台）
- 对 SEO 不敏感的场景

##### 2️⃣ SSR（Server-Side Rendering，服务端渲染）
- 请求时在服务端拉数据
- 生成完整 HTML 返回给浏览器
- 浏览器再进行 **Hydration（水合）**

**优点**
- 首屏快（HTML 有内容）
- SEO 友好

**缺点**
- TTI 较慢（要等 JS 水合）
- 服务端压力大
- 实现复杂

**适用场景**
- 首页 / 落地页
- 内容型页面
- 对 SEO 要求高的页面

##### 3️⃣ SSG（Static Site Generation，静态生成）
- **构建阶段**生成 HTML
- 部署后直接 CDN 返回静态文件

**优点**
- 性能最好（几乎没有计算）
- 成本最低
- SEO 非常好
- 稳定性强
**缺点**
- 内容更新不灵活
- 构建时间可能很长
**适用场景**
- 博客
- 文档站
- 活动页
- 内容更新不频繁的页面

##### 4️⃣ ISR（Incremental Static Regeneration，增量静态生成）
- **SSG + 按需重新生成**
- 页面先用旧静态 HTML
- 到达一定时间或触发条件后，在后台重新生成

**特点**
- 不阻塞用户请求
- 兼顾性能和内容更新

**一句话理解**： **ISR = 静态页面 + 后台悄悄更新**

**适用场景**
- 商品详情页
- 新闻列表
- 内容会变，但不要求实时

#### 对比（面试最清晰）

|模式|HTML 生成时机|首屏性能|SEO|服务器压力|适用场景|
|---|---|---|---|---|---|
|CSR|浏览器|❌ 慢|❌ 差|低|后台 / 强交互|
|SSR|请求时|✅ 快|✅ 好|高|首页 / SEO|
|SSG|构建时|🚀 极快|🚀 极好|极低|博客 / 文档|
|ISR|构建 + 后台|🚀 快|🚀 好|低|商品 / 内容站|

#### 总结：
CSR 是在浏览器端通过 JS 渲染页面，首屏慢但交互灵活；  
SSR 是在请求时由服务端生成 HTML，首屏和 SEO 更好，但需要水合，服务器压力较大；  
SSG 在构建阶段生成静态 HTML，性能和稳定性最好，但内容更新不灵活；  
ISR 是在 SSG 基础上支持增量更新，兼顾性能和内容时效性。  
实际工程中通常根据页面类型混合使用这些方案。


#### SPA-MPA

S**PA（Single Page Application）**
- 整个应用只有一个 HTML
- 页面切换通过 **前端路由**
- 数据驱动视图更新，不发生整页刷新
 ✅ 优点
- 页面切换快，体验接近 App    
- 前后端分离清晰
- 复杂交互和状态管理更友好
⚠️ 缺点
- 首屏依赖 JS，可能慢
- SEO 不友好（需 SSR/预渲染）

**MPA（Multi Page Application）**
- 每个页面都是一个独立 HTML
- 页面跳转会 **重新加载页面**
- 服务端负责页面切换
 ✅ 优点
- SEO 天然友好
- 首屏快、结构简单
 ⚠️ 缺点
- 页面跳转慢
- 状态难共享

适合 SPA
- 后台管理系统
- 复杂交互应用
- Web App（重交互、重状态）
适合 MPA
- 官网 / 营销页
- 内容型网站
- SEO 要求高的场景

|维度|SPA|MPA|
|---|---|---|
|页面结构|单一 HTML|多 HTML|
|路由方式|前端路由|服务端路由|
|页面跳转|不刷新|整页刷新|
|状态管理|前端维护|页面级，刷新即丢|
|首屏|可能慢|通常快|
|后续交互|非常流畅|每次跳转慢|
|SEO|较弱（需 SSR）|天然友好|
|架构复杂度|高|低|
